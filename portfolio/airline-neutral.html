This project was built for a renowned airlines and its newly acquired airlines. The business wanted to redesign the apps for both the airlines - so the products would have same user journey with different visual style in their respective brand skin. The app was to be redesigned for all the platforms, viz., mobile web, iOS and Android.

The Aim

Our aim was to build an airline-neutral design - a design that would be based on same structure and code for any number of sub-products and has to be consistent across all the platforms.

My Role

  • Partnered with the business analyst to understand existing user journeys and devise user journeys for the new airline.
  • Helped to build a scalable solution with uniformity and no code redundancy. For example: A third airline can now be easily converted into the parent product - the parent product is now airline neutral.
  • Carried out atomic design and object oriented design in SASS code
  • Partnered with the developers to carry out the refactoring with a strategy that helped to achieve consistent designs across all the platforms

The Process

In order to understand the concept in a better way, let’s take a car analogy.

Pick up a car

Let’s take model name called "Alpha" for the analogy.

To start, we picked up the mobile web app of the existing product and then moved on to iOS and android apps.

Inconsistent car components

Imagine a scenario in which each and every car of model Alpha is different. Each car has its own unique steering wheel, motor, etc. These inconsistencies may lead to unnecessary confusion and delays. Also, the maintenance would cost a lot.

Our product had a number of inconsistencies. We jotted down all of them. Few of them were:

  • inconsistent use of buttons for primary and secondary action,
  • different colour hex values for same colour,
  • different colours for links,
  • inconsistent design for forms,
  • inconsistent alignment issues

Fixing inconsistencies

Next step was to fix these inconsistencies.

We worked to remove redundant and inconsistent UI components. We ensured that the styles are clean and unambiguous across all the platforms. This was an important step to bring all the teams across different platforms on the same page. e.g: styles for primary and secondary action were now consistent across all the platforms.

If the car mechanic knows the model of the car and is aware of the nature of the model, he/she can confidently fix any issue depending upon his/her learning.

Extracting out car components

The car needs to be disintegrated into its sub components - brakes, window, steering wheel, etc.

After extraction of all the car components, a skeleton or the wireframe of the car is left.

Our task was to breakdown the product into components in SASS code. Since we had a huge amount of legacy code, we carried a hybrid approach of atomic design and object oriented design depending upon the need of our product.

Then, we carried out refactoring of the SASS code to extract out styles, atoms, molecules and various UI components.

Now our task of making components airline-neutral came into picture. We extracted out the structure of the product, i.e, layouts, templates, etc. - these will remain same for both the airlines.

Building a new car model

The structure of the car can now be used to build up a new model of the same car.

We gathered the components of the product and added the visual styles of the new product. In technical terms, the base code remained same - only the overrides were added, thus, keeping the code DRY.

The structure or the layout of all the UI components and the overall product remained the same.

Here comes the new model!

After putting together the components, we got our new product.
